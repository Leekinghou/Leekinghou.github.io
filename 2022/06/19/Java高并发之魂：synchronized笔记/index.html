Java高并发之魂：synchronized笔记

  • Synchronized简介：作用、地位、不控制并发的后果
  • 两种用法：对象锁和类锁
  • 多线程访问同步方法的7种情况：是否是static、Synchronized方法等
  • Synchronized的性质：可重入、不可中断
  • 原理：加解锁原理、可重入原理、可见性原理
  • Synchronized的缺陷：效率低、不够灵活、无法预判是否成功获取到锁
  • 常见面试问题：使用注意点、如何选择Lock或Synchronized等

synchronized性质

可重入（递归锁）

  1. 什么是可重入：指的是同一线程的外层函数获得锁之后，内层函数可以直接再次获取该锁。

​ （一个人排队打一份饭——不可重入，一个人排队打完一份再给老婆打一份再给小孩打一份直到不想打为止——可重入）

  1. 优点：避免死锁、提升封装性

不可中断

一旦这个锁已经被别人获得了，如果我还想获得，我只能选择等待或者阻塞，直到别的线程释放这个锁。如果别人永远不释放锁，那么我只能永远地等下去。

Synchronized的作用

✨一句话概括：采用synchronized修饰的代码可以保证在同一时刻最多只有一个线程执行该段代码，以达到保证并发安全的效果。

问题：

  1. 是怎么保证同一时刻只有一个线程执行该段代码的？
    • 加锁、释放锁
  • synchronized是最基本的互斥同步手段

不使用synchronized的后果

Synchronized的两种用法

synchronized 关键字最主要的三种使用方式：

一、对象锁

包括方法锁（默认锁对象为ths当前实例对象）和同步代码块锁（自己指定锁对象)

1. 修饰实例方法: 作用于当前对象实例加锁，进入同步代码前要获得 当前对象实例的锁

2. 修饰代码块 ：指定加锁对象，对给定对象/类加锁。synchronized(this|object) 表示进入同步代码库前要获得给定对象的锁。synchronized(类.class) 表示进入同步代码前要获得 当前 class 的锁

二、类锁

类锁指synchronized修饰静态的方法或指定锁为Class对象

只有一个Class对象：Java类可能会有很多个对象，但是只有1个Class对象。
本质：所以所谓的类锁，不过是Class对象的锁而已。

用法和效果：类锁只能在同一时该刻被一个对象拥有

3. 修饰静态方法: 也就是给当前类加锁，会作用于类的所有对象实例 ，进入同步代码前要获得 当前 class 的锁。因为静态成员不属于任何一个实例对象，是类成员（ static 表明这是该类的一个静态资源，不管 new 了多少个对象，只有一份）。所以，如果一个线程 A 调用一个实例对象的非静态 synchronized 方法，而线程 B 需要调用这个实例对象所属类的静态 synchronized 方法，是允许的，不会发生互斥现象，因为访问静态 synchronized 方法占用的锁是当前类的锁，而访问非静态 synchronized 方法占用的锁是当前实例对象锁。

总结：

  • synchronized 关键字加到 static 静态方法和 synchronized(class) 代码块上都是是给 Class 类上锁。
  • synchronized 关键字加到实例方法上是给对象实例上锁。
  • 尽量不要使用 synchronized(String a) 因为 JVM 中，字符串常量池具有缓存功能！

常见问题

多线程访问同步方法的各种具体情况——关于多线程访问同步方法的7种情况：

  1. 两个线程同时访问一个对象的同步方法

  2. 两个线程访问的是两个对象的同步方法

    线程不安全

  3. 两个线程访问的是synchronized的静态方法

    线程安全

  4. 同时访问同步方法与非同步方法

    线程不安全

  5. 访问同一个对象的不同的普通同步方法

    线程安全(因为锁加到了对象上，同一个对象上加锁)

  6. 同时访问静态synchronized和非静态synchronized方法

    线程不安全

  7. 方法抛出异常后，会释放锁吗

    会，先抛出异常，第一个线程释放锁，第二个线程拿到锁

其他：

  1. 使用synchronized关键字的注意点有哪些？

    所得范围不宜过大，要避免锁的嵌套

  2. 如何选择Lock和synchronized关键字？

    看需求：Lock需要手动释放锁

  3. 如何提升synchronized的性能？

    读多写少时：Synchronized性能较差是因为它使得同时只有一个线程可以执行，可以使用读写锁

    需要灵活：Synchronized性能较差是因为释放锁的时机被规定死了，使用Lock可以灵活控制和释放锁

  4. 谈谈 synchronized 和 ReentrantLock 的区别

    1. 前者获得和释放锁写死，后者是手动决定获得与释放锁
    2. 两者都是可重入锁
    3. synchronized 依赖于 JVM 而 ReentrantLock 依赖于 API
      • ReentrantLock 是 JDK 层面实现的（也就是 API 层面，需要 lock() 和 unlock() 方法配合 try/finally 语句块来完成）
    4. ReentrantLock 比 synchronized 增加了一些高级功能：
      • 等待可中断
      • 可实现公平锁
        • ReentrantLock可以指定是公平锁还是非公平锁。而synchronized只能是非公平锁。所谓的公平锁就是先等待的线程先获得锁。ReentrantLock默认情况是非公平的，可以通过 ReentrantLock类的ReentrantLock(boolean fair)构造方法来制定是否是公平的
      • 可实现选择性通知（锁可以绑定多个条件）
        • synchronized关键字与wait()和notify()/notifyAll()方法相结合可以实现等待/通知机制。
        • ReentrantLock类需要借助于Condition接口与newCondition()方法
      • 如果你想使用上述功能，那么选择 ReentrantLock 是一个不错的选择。性能已不是选择标准

原理

  1. 加锁和释放锁的原理：看字节码
  2. 可重入原理：加锁次数计数器
  3. 保证可见性的原理：内存模型

加锁和释放锁的原理

  1. 获取和释放锁的时机：进入和退出同步代码块（包括抛出异常）

  2. 代码原理（等价表示）

  3. 看monitor指令

    加锁即时monitorenter，释放锁即是monitorexit，第二个monitorexit是异常退出的释放锁。

可重入原理

  • 可重入原理：加锁次数计数器
    1. JVM会记录被加锁的次数
    2. 第一次加锁时，次数从0变为1，之后如果再次加锁，就从1变成2， 以此类推
    3. 退出一层同步代码块时，计数减一，当计数为0的时候，代表锁释放

可见性

  1. 一个线程执行的结果，另外的线程不一定可见
  2. 线程1操作x=5,之后线程2可能读取x=3
  3. synchronized是如何做到可见性的实现的：某个被synchronized修饰的方法或者代码块在执行完毕后，该方法或者代码块中对共享变量的所作的任何修改，都要在释放锁之前从线程内存写入到主内存中，因此就保证了线程内存的变量与主内存中变量的一致性。同样，在进入同步代码块或者方法中所得到的共享变量值也是直接从主内存中获取的。

可见性及可见性图片来自https://blog.51cto.com/u_3664660/3216492

Synchronized关键字的缺陷

  • 效率低

    1. 锁的释放情况少（执行完、发生异常）
    2. 试图获得锁时不能设定超时（不撞南墙不回头）
    3. 不能中断一个正在试图获得锁的线程

  • 不够灵活（读写锁更灵活）

    1. 加锁和释放的时机单一，每个锁仅有单一的条件（某个对象)，可能是不够的

  • 无法获知是否成功获得锁

总结

JVM会自动通过使用monitor来加锁和解锁，保证了同时只有一个线程可以执行指定代码，从而保证了线程安全，同时具有可重
入和不可中断的性质。
